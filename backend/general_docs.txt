1. Цели и границы проекта
•	Создать веб-платформу, на которой правообладатель может зарегистрироваться, пройти идентификацию, подтвердить связь с объектом интеллектуальной собственности и инициировать токенизацию актива.
•	Обеспечить выпуск токенизированного актива в сети Solana после прохождения проверок и внутреннего approval workflow.
•	Предоставить маркетплейс, на котором инвесторы могут просматривать активы , проходить onboarding и покупать токены за SOL.
•	Соблюдать принцип: чувствительные действия (токенизация, покупка, листинг, вывод средств) доступны только после обязательных compliance-проверок; KYC может быть необязательным на этапе просмотра и предварительной подачи заявки, но не на этапе выпуска и торговли.
•	KYC и верифицировании нужно только при подаче на патент
•	Монетизация происходит через комисии с операций покупки токенов
•	Профит для владельцов патентов получать первичные инвестиции
•	Профит для инвесторов от дивидентов(дивиденты фиксированные и обязательно платиться с кошелька патентова )
•	Будут фиксированное длинны временный выплат дивидентов и обязательно выплачиваються с кошелька владельца патента, если не получаються выплаты то идет система предупржедений
•	Если более 2 предупреждений то проект закрываеться и все дивиденты возращаються
2. Предположения и ограничения
•	Проект стартует как MVP с упором на один тип объектов: патенты США.
•	Проверка патента выполняется через внешние API и реестры USPTO; юридическое решение о допустимости токенизации подтверждается внутренним manual review.
•	Персональные данные, изображения документов и результаты KYC не размещаются on-chain.
•	On-chain слой хранит только необходимые данные о выпуске токена, адресах mint/account и результатах транзакций.
•	Платформа проектируется как permissioned marketplace: выпуск и покупка активов контролируются backend-логикой и compliance-правилами.

4. Функциональные блоки платформы
4.1. Identity & Access
Регистрация, авторизация, phone\otp verification, роли, управление сессиями, привязка wallet address.
4.2. KYC / Compliance
Запуск и мониторинг KYC-процесса, получение webhook-результатов, sanctions/risk screening, ручной review спорных кейсов.
4.3. IP Verification
Подача: patent number; ФИО; адресс; удостверение pdf; видеокружоек, запрос к USPTO API, сопоставление с данными заявителя, проверка supporting documents, решение о допустимости токенизации.
4.4. Asset Management
Создание карточки актива, хранение метаданных, статусов, юридических документов, состава выпуска, параметров токенизации.
4.5. Tokenization
Mint токена в сети Solana, настройка supply, treasury и адресов хранения, привязка on-chain сущностей к off-chain записям актива.
4.6. Marketplace
Листинг активов, размещение офферов, покупка токенов за SOL, контроль допустимости сделки и пост-обработка settlement.
4.7. Admin & Audit
Панели управления, отчётность, хронология действий, журнал ошибок, экспорт данных и мониторинг интеграций.Создание карточки актива, хранение метаданных, статусов, юридических документов, состава выпуска, параметров токенизации.

6. Архитектура базы данных
Рекомендуемая СУБД: PostgreSQL (MVP допускает SQLite через aiosqlite). Доступ к данным — через SQLAlchemy 2.0 Async ORM с миграциями Alembic. Для документов и вложений используется отдельное объектное хранилище (S3-compatible storage на продакшене, локальная папка uploads/ в разработке).

6.1. Таблицы (логические группы)

Домен пользователей (app/models/user.py)
• users — основная модель: id (UUID PK), email (unique, indexed), password_hash (nullable для инвесторов), role (enum: user/issuer/investor/compliance_officer/admin), status (enum: pending_otp/active/suspended/blocked/rejected)
• profiles — профиль пользователя: user_id (PK, FK → users.id), full_name, country, organization_name, preferred_language
• kyc_cases — KYC-кейсы: id (UUID PK), user_id (FK), provider, provider_case_id, status (enum KYCCaseStatus), risk_level (enum KYCRiskLevel), webhook payloads
• wallet_links — привязка кошельков: id (UUID PK), user_id, wallet_address, network (Solana), is_primary. UniqueConstraint(wallet_address, network)
• otp_codes — OTP-коды: id (UUID PK), user_id, code (6 цифр), purpose, is_used, expires_at
• verification_cases — верификация для patent submitters: id (UUID PK), user_id, patent_name, patent_address, user_address, id_document_url, selfie_url, status, reviewer_notes

Домен патентов (app/models/patent.py)
• patents — патенты: id (UUID PK), owner_user_id (FK → users.id), patent_number, jurisdiction, title, abstract, status (enum: draft/submitted/prechecked/under_review/approved/rejected/archived)
• patent_documents — документы патента: id (UUID PK), patent_id (FK), document_type, storage_key, created_by_user_id
• patent_reviews — рецензии: id (UUID PK), patent_id (FK), reviewer_user_id (FK), decision (enum ReviewDecision), notes, reviewed_at

Домен IP-claims (app/models/ip_claim.py)
• ip_claims — заявки на IP: id (UUID PK), issuer_user_id (FK), patent_number, patent_title, claimed_owner_name, description, jurisdiction, status (enum: draft/submitted/prechecked/awaiting_kyc/under_review/approved/rejected), prechecked (bool), precheck_status, source_id, checked_at, patent_metadata (JSON), external_metadata (JSON)
• ip_documents — документы заявки: id (UUID PK), ip_claim_id (FK), file_url, doc_type, created_by_user_id
• ip_reviews — рецензии на claims: id (UUID PK), ip_claim_id (FK), reviewer_id (FK), decision (enum: approve/reject/request_more_data), notes
• token_revocations — отозванные токены: jti (PK), token_type, expires_at, revoked_at

Кэш IP Intelligence (app/models/ip_intel.py)
• patent_cache — кэш патентных данных: id (UUID PK), source (enum: USPTO/PATENTSVIEW/EPO_OPS/WIPO_PCT), source_id, country_code, patent_number, kind_code, title, abstract, filing_date, publication_date, grant_date, status, JSON-поля (assignees, inventors, cpc_classes, uspc_classes, ipc_classes, geo_data, family_ids, raw_data), cached_at, expires_at, ip_claim_id (FK)
• patent_search_cache — кэш поисковых запросов: id (UUID PK), query_hash (SHA256, indexed), query_text, countries, date_from, date_to, results (JSON), total_count, sources_queried, cached_at, expires_at

Общие таблицы (app/models/common.py)
• audit_logs — аудит действий: id (UUID PK), actor_id (FK → users.id, nullable), action, entity_type, entity_id, payload (JSON без PII), created_at
• webhook_events — входящие вебхуки: id (UUID PK), source, external_id, status (enum: received/processed/failed/dead_letter), payload (JSON), created_at, updated_at

Аналитика (app/models/analytics.py)
• user_metrics_daily — ежедневные метрики: date (PK), total_users, new_users, active_users, kyc_started, kyc_approved
• patent_metrics_daily — метрики патентов: date (PK), total_patents, new_patents, patents_under_review, patents_approved, patents_rejected
• kyc_funnel_stats — KYC воронка: composite PK (period_start, period_end, step_name), step_name (enum KYCFunnelStep), users_count

6.2. Логические связи
• Один пользователь имеет одну запись users, один профиль profiles и несколько кошельков в wallet_links.
• Один пользователь может иметь несколько KYC-кейсов (повторные проверки), но только один активный кейс со статусом pending/manual_review.
• Одна заявка ip_claims относится к одному issuer, но может иметь несколько supporting documents и одну или несколько записей ручной проверки (ip_reviews).
• Один актив assets создаётся на базе одобренной заявки ip_claims и имеет максимум один актуальный token_issuance на MVP-этапе.
• Один listing относится к одному asset; один listing может порождать множество orders и trades.

7.2. Слой API (endpoints)
• Тонкие контроллеры: валидация входных данных через Pydantic, вызов сервисов, возврат response
• Авторизация через JWT Bearer token + role-based Depends(require_roles(...))
• Dependency Injection: AsyncSession через get_db, текущий пользователь через get_current_user

7.3. Слой сервисов (business logic)
• UserService — CRUD пользователей, хэширование паролей (PBKDF2-HMAC-SHA256, 260,000 iterations), профили, admin CRUD (list_users, get_user_admin_detail, admin_update_user, change_user_status, soft_delete_user)
• AuthService — JWT token management (access/refresh/otp/password_reset), revocation, cleanup
• OtpService — ДВА режима работы:
  1. Redis-based (новый API): generate_and_send_otp(), verify_otp()
     - OTP хранится в Redis как HMAC-SHA256 хэш с TTL 300 секунд
     - Ключ: otp:{purpose}:{normalized_identifier}
     - Классификация идентификатора: email (содержит @) или phone (E.164, начинается с +)
     - Доставка: email через SMTP (smtplib), SMS через stub (NotImplementedError)
     - Максимум 5 попыток верификации, затем блокировка
     - При успешной верификации возвращается verified_token (JWT, 10 мин)
  2. SQLAlchemy-based (legacy API): OTPService.create_otp(), OTPService.verify_otp()
     - OTP хранится в таблице otp_codes (БД)
     - Используется старыми endpoints /otp/send и /otp/verify
• PatentService — precheck патентов (MVP: deterministic adapter), запросы к внешним API
• AdminPatentService — admin CRUD для патентов (list_patents_admin, get_patent_admin_detail, change_patent_status)
• IpClaimService — CRUD заявок, загрузка документов, review workflow
• IpIntelService — оркестрация IP Intelligence: кэш-проверка → внешний API → нормализация → обогащение
• AuditService — запись аудит-логов для критичных операций

7.4. Слой интеграций (external APIs)
• BasePatentClient (abstract) — retry (tenacity, exponential backoff 2–30s), rate limiting, audit logging
• UsptoPatentClient — USPTO Open Data Portal v2.3 (data.uspto.gov)
  - Endpoints: GET /api/v1/patent/grants, POST /api/v1/patent/applications/search, GET /api/v1/patent/applications/{id}/continuity
  - Auth: API Key в X-API-Key, 5 calls/sec, cache 72h
  - Стратегия: grants → fallback applications → continuity для статуса
  - Нормализация: CPC/USPC/IPC классы, assignees, inventors, status mapping
• PatentsViewClient — без авторизации, 10 calls/sec, cache 48h
• EpoOpsClient — OAuth2 Bearer (TODO: client credentials flow), 5 calls/sec, cache 48h
• WipoPatentscopeClient — API Key (partial implementation), cache 48h
• OtpSender — email (SMTP) и SMS delivery:
  - send_email_otp(): SMTP с STARTTLS, env vars: SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS
  - send_sms_otp(): STUB для будущей интеграции (Firebase Auth / MSG91 / Vonage)

7.5. Аутентификация и авторизация
• Access Token: JWT, 30 мин TTL, payload: sub (email), type, exp, iat, jti
• Refresh Token: JWT, 7 дней TTL, при logout jti записывается в token_revocations
• OTP Token: JWT, 10 мин TTL, для верификации
• Verified Token: JWT, 10 мин TTL, возвращается после успешной верификации OTP (otp-verify endpoint)
• Password Reset Token: JWT, 30 мин TTL
• Роли: user / issuer / investor / compliance_officer / admin
• Guard: @router.get("...", dependencies=[Depends(require_roles("admin"))])
• OTP Delivery Architecture:
  - Redis-based: otp:{purpose}:{identifier} → {"otp_hash": hmac_sha256, "attempts_left": 5, "expires_at": timestamp}
  - HMAC-SHA256 хэширование кодов (env var: OTP_HMAC_SECRET)
  - Идентификаторы нормализуются: email → lowercase + strip, phone → strip only
  - Rate limiting: 3 повторных отправки на 10 минут (Redis counter)
  - SMS provider: STUB (TODO: Firebase Auth / MSG91 / Vonage)
  - Email provider: SMTP (smtplib, STARTTLS)

11. Нефункциональные требования и безопасность
•	PII хранится только в защищённой базе/хранилище; доступ ограничивается по ролям.
•	Пароли хранятся только в виде secure hash (PBKDF2-HMAC-SHA256, 260,000 iterations).
•	OTP коды хранятся ТОЛЬКО в Redis в виде HMAC-SHA256 хэшей, никогда не возвращаются в API ответах.
•	OTP_HMAC_SECRET — критичная переменная окружения, генерируется через secrets.token_hex(32).
•	Доступ к административным и review-маршрутам защищён RBAC и audit trail.
•	Внешние webhooks проверяются по подписи и идемпотентности.
•	Ошибки интеграций не должны приводить к потере данных; обязательна очередь повторов и dead-letter обработка.
•	On-chain приватные ключи сервиса не хранятся в исходном коде; используются secrets manager / защищённые env.
•	Все критичные операции (approve, mint, settle, reject) логируются в audit_logs.
•	SMS provider — STUB, реальная интеграция планируется (Firebase Auth / MSG91 / Vonage).
•	Email delivery — SMTP с STARTTLS, App Password для Gmail (требуется 2FA).

Итоговая рекомендация
Начинать проект следует как compliance-first платформу: off-chain верификация, KYC и review составляют источник истины, а on-chain слой Solana используется для controlled issuance и расчётов. Для MVP рекомендуется ограничить scope одним типом IP, одним KYC-провайдером, одним сценарием токенизации и упрощённым маркетплейсом.
